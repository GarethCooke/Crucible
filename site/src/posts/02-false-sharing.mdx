Same algorithm. Same fill stream. Same machine. Two versions of a single struct — the
only difference is 56 bytes of padding. One collapses as thread count grows; the other
scales near-linearly. The IPC counter drops in lockstep with the throughput collapse,
and the cache-miss counter rises to match. Whether that collapse costs you wall-clock
time depends on where the coherency traffic has to travel.

## The setup

Each thread owns a slot in a shared array of P&L accumulators. The benchmark repeatedly
walks a pre-generated fill stream and adds each value to the thread's slot. All threads
run concurrently, targeting different slots — there is no logical sharing of data, only
incidental sharing of hardware.

<CodeCompare
  lang="cpp"
  labels={["Unpadded", "Padded"]}
  naive={`// Adjacent slots share cache lines.
// 8 bytes per slot → 8 slots per 64-byte line.
struct Strategy {
    volatile double pnl;
};

Strategy strategies[8];  // threads 0–7`}
  optimized={`// Each slot owns its cache line.
// alignas(64) guarantees no neighbours.
struct alignas(64) PaddedStrategy {
    volatile double pnl;
    char pad[64 - sizeof(double)];
};

PaddedStrategy strategies[8];`}
/>

The fill stream is 1,024 doubles (8 KB), sized to sit comfortably in the 32 KB L1d
cache. It is generated once at startup with a fixed seed and shared read-only across
all threads, so every thread reads identical data — the only writes are the per-slot
`pnl` accumulations.

## The counter signal — intra-CCX

The clearest evidence of false sharing is in the performance counters. With all threads
pinned to one CCX (cores 4–7), watch IPC and cache miss rate as thread count grows:

<CounterOverlay slug="false-sharing-pnl" placement="intra-ccx" metric="cache_miss_ratio" />

At 1 thread, both layouts are identical — no contention, near-zero miss rate, similar
IPC. At 4 threads, the unpadded variant's IPC falls to **0.11** while the padded
variant holds at **0.56** — a **5× IPC collapse**. The cache miss rate tracks the same
story: unpadded goes from ~2.5% at 1 thread to **~80% at 4 threads**; padded stays
near 2.8% throughout.

What is happening: every write to an unpadded slot invalidates the entire cache line,
which holds seven neighbouring slots belonging to other threads. Each thread must
re-fetch the line from the L3 before it can write. The accesses that previously
resolved in the L1d (~4 cycles) now require L3 round-trips (~35 cycles on Zen 2), and
with four threads simultaneously invalidating each other, the CPU spends most of its
time waiting rather than executing.

## Wall-clock impact — intra-CCX

The counter signal is the reliable part. The wall-clock impact within one CCX is muted
on Zen 2:

<ThroughputBars slug="false-sharing-pnl" placement="intra-ccx" />

At 4 threads, the unpadded variant is noticeably slower than the padded variant — but
the ratio is around 1.2×, far below the ~5× the counter collapse would suggest. The
reason: all four cores share the same 16 MB L3 cache. Coherency traffic between cores
in the same CCX resolves within that shared cache at tens-of-nanoseconds per hop,
without touching DRAM. The hardware absorbs the penalty; the counters see it; the
wall clock mostly doesn't.

## Where wall-clock magnification reasserts itself

Zen 2 is a chiplet architecture: two 4-core CCXs on a shared memory controller die,
connected via AMD's Infinity Fabric. Coherency traffic between two cores on the same
CCX travels through the shared 16 MB L3. Coherency traffic between CCX0 and CCX1 must
cross the Infinity Fabric — higher latency, no shared cache to absorb the round-trips.

With threads distributed across both CCXs, the counter collapse looks the same — false
sharing is firing identically — but the wall-clock penalty now has somewhere new to land:

<CounterOverlay slug="false-sharing-pnl" placement="cross-ccx" metric="instructions_per_cycle" />

<ThroughputBars slug="false-sharing-pnl" placement="cross-ccx" />

The counter signal at 2t cross-CCX is nearly identical to 2t intra-CCX: IPC ~0.17 in
both cases, miss rate ~68% vs ~80%. The mechanism is the same. But the wall-clock cost
is surprisingly restrained.

Conventional wisdom for AMD Zen 2 says cross-CCX cache-line transfer is 2–3× slower
than intra-CCX — about 80 ns vs 25 ns for an isolated line fetch. You might extrapolate
that false sharing across the Infinity Fabric will be similarly punishing.

It isn't. On the 3800X, the cross-CCX penalty for false sharing is a remarkably
consistent **1.28×** at both 2 and 4 contending threads, with CV under 0.25% across 11
repetitions.

The reason is dilution. Each loop iteration does the coherence ping-pong _plus_ some
local work, and the local work doesn't care which CCX the other thread sits on. That
uncontended work absorbs some of the IF round-trip latency on every iteration.
Back-of-envelope: with around 200 ns of local work per iteration,
`(80 + 200) / (25 + 200) ≈ 1.24×` — within rounding of what we measure.

| Test                 | Median (ns) | Items/s    |
| -------------------- | ----------- | ---------- |
| IntraCCX/2t/unpadded | 7,761,535   | 188.98 G/s |
| CrossCCX/2t/unpadded | 9,956,942   | 196.43 G/s |
| IntraCCX/4t/unpadded | 14,792,861  | 415.50 G/s |
| CrossCCX/4t/unpadded | 18,862,904  | 368.12 G/s |

Two thread counts are reported because they carry different signals. At 4 threads the cache line is hot enough that Infinity Fabric latency dominates any ambient noise, and the measured 1.31× ratio is reproducible across separate runs to within 1%. At 2 threads the same physical effect is present at a similar magnitude, but the measurement is more sensitive to whatever else cpu 0 happens to be doing during the run — the reference machine uses `cset shield` for isolation rather than `isolcpus=` boot params (see [methodology](/methodology)), and `cset` is best-effort. We treat the 4t number as the definitive cross-CCX result; 2t is reported for completeness but is not used as a regression sanity check.

At 8 threads cross-CCX, the unpadded variant reaches **68.2 ns/op** against the padded
variant's **0.20 ns/op** — a **341× gap**. The padded 8-thread variant shows near-perfect
linear throughput scaling (8× vs 1 thread).

## Sharing is what costs you, not crossing the Fabric

The padded baseline closes the loop. With padding, each thread owns its own cache line
— there is no coherency traffic to send across the Infinity Fabric. The cross-CCX wall
overhead collapses to ~1.0×.

The full penalty stack on a 3800X:

| Configuration       | Wall-time, relative                            |
| ------------------- | ---------------------------------------------- |
| Padded, intra-CCX   | 1.0× (baseline)                                |
| Padded, cross-CCX   | ~1.0× (IF invisible without sharing)           |
| Unpadded, intra-CCX | ~5× (false sharing within CCX)                 |
| Unpadded, cross-CCX | ~5× × 1.28× ≈ 6.4× (false sharing across CCX) |

Design your data layout so threads don't share lines. Once you've solved that
first-order problem, whether the threads happen to land on the same CCX or not is a
second-order concern.

## The regime question

False sharing has two separable effects:

**Counter collapse is universal.** Wherever threads share a cache line, the
invalidation/refetch cycle fires. IPC drops. Miss rate climbs. This is visible
regardless of topology and regardless of whether wall-clock time degrades. If your
profiler shows an unexplained IPC collapse under thread scale, false sharing is on the
list of suspects.

**Wall-clock penalty depends on coherency-domain topology.** Modern shared-LLC
architectures — Zen 2 within a CCX, Intel within a socket — absorb much of the cost
in the shared cache. The round-trip happens, but it stays in silicon at tens-of-ns.
Cross-coherency-domain transfers — cross-CCX on Zen 2, cross-socket, NUMA — remove
that cushion. The same false-sharing pattern that causes a ~1.2× wall-clock regression
intra-CCX causes a **~1.28×** regression from crossing the Infinity Fabric — modest,
because the non-contended work per iteration dilutes the IF round-trip. The full
unpadded-vs-padded gap at 8 threads cross-CCX reaches **341×**.

This has a practical implication for measurement: benchmarking false sharing on a
single socket may understate the production penalty if the production system spans NUMA
nodes, or runs on a processor with discrete CCX/CCD partitioning.

## What this means in practice

False sharing is easy to introduce accidentally in systems with per-thread state:

- **Per-strategy accumulators in trading engines** — exactly the pattern modelled here.
  A common layout mistake is `double pnl[N_STRATEGIES]` with one thread per strategy.
- **Per-worker statistics counters** — hit counts, error rates, queue depths. If each
  counter is a single `int64_t` in a contiguous array, adjacent counters share lines.
- **Market-data fan-out structs** — if writers update sequence numbers on the same
  cache line as fields read by consuming threads, readers stall on coherency traffic
  that has nothing to do with their data.

The fix is straightforward: align each per-thread slot to the cache-line size.
`alignas(64)` is sufficient on x86-64 (and most ARM64 deployments). The trade-off is
memory: 8 threads × 64 bytes = 512 bytes instead of 64 bytes. For hot-path accumulators
the trade-off is almost always worthwhile.

What this benchmark doesn't show is subtler cases: false sharing between a writer thread
and a reader thread on the same cache line, or producer–consumer patterns where the
sequence number and the payload are laid out on the same line. The magnitude of those
effects depends on access frequency and temporal overlap — but the mechanism and the fix
are identical.

## A note on measurement conditions

Core isolation uses `cset shield --cpu=0-7`, invoked by the wrapper script immediately
before each benchmark run and reset after (see [methodology](/methodology)). `cset` shields
all 8 cores, but cpu 0 sits on CCX0 alongside the system timer and other unmovable kernel
threads — it sees more ambient noise than the CCX1 cores. CV across 11 reps was
0.02–0.22% intra-run, which confirms that background noise isn't materially affecting
single-run measurements. Across separate runs, the CrossCCX/2t result is more variable
than 4t for exactly this reason — which is why we treat the 4t number as the definitive
cross-CCX result.

## Reproducing this

Source files committed to the repo:

- Benchmark: [bench/demos/02-false-sharing/false_sharing_pnl.cpp](bench/demos/02-false-sharing/false_sharing_pnl.cpp)
- Capture script: [tools/perf_capture.sh](tools/perf_capture.sh)
- Data parser: [tools/parse_perf.py](tools/parse_perf.py)

Per-variant capture (example — intra-CCX, 4 threads, unpadded):

```bash
# Build
cd bench && cmake -B build -DCMAKE_BUILD_TYPE=Release && cmake --build build --parallel

# Verify store is not elided (see comment at top of false_sharing_pnl.cpp)
objdump -d build/demos/02-false-sharing/bench_02_false_sharing_pnl \
  | grep -A 40 "<_ZL9worker_fn" | grep "movsd"

# Capture (one call per variant; repeat for all 12)
./tools/perf_capture.sh \
  build/demos/02-false-sharing/bench_02_false_sharing_pnl \
  intra-ccx 4 0   # placement threads padded(0/1)

# Fold into JSON
python3 tools/parse_perf.py \
  --perf  intra-ccx_4t_unpadded.perf.json \
  --bench intra-ccx_4t_unpadded.bench.json \
  --out   site/src/data/perf/false-sharing-pnl.json \
  --placement intra-ccx --threads 4 --padded 0
```

For the L1D-specific miss counter, confirm the event name on Zen 2 before capture:

```bash
perf list | grep -i l1d   # look for l1d.replacement or L1-dcache-load-misses
```

Pre-flight: SMT must be off (`cat /sys/devices/system/cpu/smt/active` → `0`),
governor = performance, turbo off, `cpuset` package installed.
The benchmark binary aborts with a clear error message if SMT is active.
Cross-CCX runs use non-isolated cores (cores 0 and 4); see the `notes` field in the
JSON for the caveat.

To verify thread-to-CPU pinning (invaluable when debugging topology issues), set the
`CRUCIBLE_PRINT_AFFINITY` environment variable before running:

```bash
CRUCIBLE_PRINT_AFFINITY=1 ./bench/build/demos/02-false-sharing/bench_02_false_sharing_pnl \
  --benchmark_filter='CrossCCX/2t/unpadded' \
  --benchmark_repetitions=1 2>&1 | grep '\[bench\]'
# Expect: one cpu in {0,1,2,3} and one in {4,5,6,7}
```
