# Crucible — Demo 09 plan

Tracking doc for demo 9 (ARM NEON cross-arch: Black-Scholes call pricing on the Raspberry Pi 5, the cross-arch companion to demo 3's SIMD shootout on Zen 2). The teaser stub ships to `main` first (§0); substantive work proceeds on a feature branch. This is the first demo on a **second reference machine** (Pi 5, BCM2712, Cortex-A76, AArch64), so it carries a rig bring-up step (§1) that earlier demos didn't need.

All demo 9 briefs referenced in this plan live in `docs/briefs/09/`; filenames below are given bare and resolve to that folder.

This plan applies the lessons from demos 5–8:

1. **Preflight calibration is a task, and the pilot may reframe the thesis.** Demo 9's headline is pilot-contingent: it depends on the NEON-over-scalar ratio landing near the 4-wide ceiling (~4×) at a cache-resident N, and on whether hand-written NEON beats the autovectoriser. The pilot (§2) confirms both before the brief commits, and decides whether the post is one story (the width ceiling alone) or two.
2. **Capture environment is explicit — and is a new machine.** Headless boot, single isolated core (core 3 via `taskset`), matching demos 1–8 in intent but with Pi-appropriate mechanisms. The Pi's "turbo off, verified" is a pinned clock (min=max at 2400 MHz) plus `get_throttled=0x0`, not the Zen 2 `CRUCIBLE_TURBO=off`. Stated in the brief, not assumed.
3. **`cset shield` is NOT used on the Pi.** Bookworm is cgroup v2; `cset` targets v1. Isolation here is `isolcpus=2,3` (boot) + `taskset -c 3` (capture) + IRQ affinity (`pi-rig-setup.sh`). The per-capture precondition is a green `pi-preflight.sh`, not `cset shield --reset`.

## Story angle (provisional — pilot confirms)

Not "SIMD also works on ARM" — demo 3 already proved vectorisation. The reason to ship is architectural honesty about **vector width**, and the proof that the methodology travels across ISA and microarchitecture.

- **Headline — the width ceiling.** The Cortex-A76 is NEON-only: 128-bit, 4-wide float32, same width as SSE, **no SVE**. So the Black-Scholes kernel tops out at ~4× over scalar and _stays there_ — there is no wider unit to reach for, where the Zen 2 box went on to AVX2 (which itself only realised ~2.0–2.25× over SSE because Zen 2 cracks 256-bit into 2×128-bit µops). The punchline: vector width is an architectural property you don't get to assume. A well-told ceiling, on-brand for the hype-allergic reader, and it cross-links cleanly to demo 3.
- **Secondary (pilot-contingent) — hand-NEON vs the autovectoriser.** On AArch64 NEON is baseline, so `-O3` autovectorises by default — unlike x86, where SIMD is opt-in. If hand-written intrinsics still beat GCC's autovec by a chartable margin, that's a second axis; if autovec already matches, the axis collapses and the demo rests on the ceiling alone. Either outcome is honest; §2 decides.

**What the post does NOT claim:** that NEON is faster or slower than AVX2 in absolute terms — **cross-machine absolute ns/op is never compared** (different clocks, nodes, memory; the only portable quantity is the within-machine speedup ratio); that ARM is "slow" or "fast"; that the result generalises to all of ARM (it's A76-specific — a core _with_ SVE would scale past the ceiling, which is a possible sequel, not this post).

## Lessons from demos 5–8 applied to this plan

- **§2 exists at all, and can collapse the secondary story.** The headline number (≈4× ceiling) and the one-story-or-two question are both pilot-contingent. If A2 comes in sub-ceiling, the demo re-scopes before §3 commits rather than shipping a number the hardware won't give.
- **Cross-machine honesty is baked in, not bolted on.** Absolute ns/op is never compared across the two rigs; the within-machine speedup ratio is the only portable quantity. Named in the story angle, §6, §7, and checked explicitly in §9.
- **A second reference machine is a methodology event.** The machine-block schema was written for the single Zen 2 rig; representing the Pi may need a field (`arch`/`soc`). Surfaced in §4 and the methodology-page update in §8.
- **ARM porting traps named up front.** Denormal flush is the FPCR FZ bit (not x86 `_MM_SET_FLUSH_ZERO_MODE`), and the scalar baseline needs an explicit anti-autovec guard since NEON is on by default at `-O3`. Both are in the pilot bench brief and re-checked in §5.
- **Reuse charts, don't fork.** TimeVsN / ThroughputBars / CounterOverlay / CodeCompare all carry over; the cross-arch ratio view is a composition decision, not a new component.
- **No backlog brief-format change.** Demo 7/8's shape worked. Keeping it.
