# Crucible — demo 01 disassembly guard fix (run_one.sh)

CC brief, branch `fix/turbo-off-recapture`. The demo-01 disassembly guard in `bench/scripts/run_one.sh` (~lines 598–640) fails the current honest build. Investigation verdict: **the compiled branch is intact and byte-identical to the original capture's inner loop** (`cmp $0x7f,%edx; jle`); the guard itself has two defects and has never correctly verified the function. Fix the guard only — the benchmark source and CMake flags are correct as-is.

## Context — verified on the rig

- The branchy `_ZL13sum_thresholdRKSt6vectorIiSaIiEE` compiles to a genuine conditional branch: `cmp $0x7f,%edx` / **`jle`**. Old (committed `01-branch-prediction.disasm.txt`) and new builds have identical inner loops; only `endbr64`, nop padding, and addresses differ.
- **Defect 1 — mnemonic list:** the guard requires `\b(jl|jge|jb|jae)\b`. The actual instruction is `jle`, which matches none of these (`\bjl\b` does not match `jle`). The list never contained the mnemonic the function actually uses.
- **Defect 2 — leaky extraction:** `awk '/<_ZL13sum_threshold[^_]/,/^$/'` re-opens the range on every line *referencing* the symbol (call sites, `lea` comment annotations), sweeping unrelated code chunks into the extraction. The old committed disasm file (465 lines) contains such swept-in code; the guard historically passed because a `jl`-family instruction existed somewhere in that noise — a false pass. The new build's swept-in chunks contain none, exposing Defect 1.

## Tasks

### Task 1 — anchor the branchy extraction to the function header
objdump function headers start at column 0 and end in `>:`; reference lines are indented and end in `>`. Find:
```
        | awk '/<_ZL13sum_threshold[^_]/,/^$/' \
```
Replace with:
```
        | awk '/^[0-9a-f]+ <_ZL13sum_threshold[^_].*>:/,/^$/' \
```

### Task 2 — correct the mnemonic check to the comparison family
Find:
```
    if ! grep -qE '\b(jl|jge|jb|jae)\b' "${DISASM_OUT}"; then
```
Replace with:
```
    if ! grep -qE '\bj(l|le|g|ge|b|be|a|ae)\b' "${DISASM_OUT}"; then
```
Deliberately **exclude `je`/`jne`** — those are loop-control/emptiness branches present even in branchless code; including them would make the check vacuous. The set above is the signed/unsigned comparison family, which is what a data-dependent threshold branch compiles to (`jle` today; tuning may flip it to `jl`/`jge` etc. on other compiler versions — all acceptable).

Update the two echo strings that name the old list (`must contain jl/jge` at ~line 604 and `no jl/jge/jb/jae found` at ~609) to say `a conditional comparison branch (jl/jle/jg/jge/jb/jbe/ja/jae)`.

### Task 3 — anchor the branchless extraction the same way
Find:
```
        | awk '/<_ZL24sum_threshold_branchless/,/^$/' 2>/dev/null || true)
```
Replace with:
```
        | awk '/^[0-9a-f]+ <_ZL24sum_threshold_branchless.*>:/,/^$/' 2>/dev/null || true)
```
(Same leak: its cmov-present / SIMD-absent checks could otherwise false-pass or false-fail on swept-in neighbouring code.)

### Task 4 — regenerate and sanity-check the committed disasm artifact
Run the guard path once (`run_one.sh 01-branch-prediction` up to/through the disassembly step, or extract equivalently) so `site/src/data/perf/01-branch-prediction.disasm.txt` is regenerated by the fixed extraction. The regenerated file legitimately differs from the committed one (`endbr64`, padding, addresses — cosmetic toolchain drift; the loop body is identical).
