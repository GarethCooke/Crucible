# Crucible ŌĆö disassembly guard fixes, demos 01 & 02

CC brief, branch `fix/turbo-off-recapture`. Two disassembly guards have the same defect family: the demo-01 guard in `bench/scripts/run_one.sh` (~lines 598ŌĆō640), which fails the current honest build, and the demo-02 guard `bench/scripts/check_volatile_codegen.sh`, which has the same leak plus a window fragility (not yet triggered, but more likely on the new toolchain). Investigation verdict for demo 01: **the compiled branch is intact and byte-identical to the original capture's inner loop** (`cmp $0x7f,%edx; jle`); the guards themselves are defective and have never correctly verified their functions. Fix the guards only ŌĆö benchmark sources and CMake flags are correct as-is. (These three objdump sites are the only disassembly checks in the pipeline ŌĆö confirmed by exhaustive grep.)

## Context ŌĆö verified on the rig

- The branchy `_ZL13sum_thresholdRKSt6vectorIiSaIiEE` compiles to a genuine conditional branch: `cmp $0x7f,%edx` / **`jle`**. Old (committed `01-branch-prediction.disasm.txt`) and new builds have identical inner loops; only `endbr64`, nop padding, and addresses differ.
- **Defect 1 ŌĆö mnemonic list:** the guard requires `\b(jl|jge|jb|jae)\b`. The actual instruction is `jle`, which matches none of these (`\bjl\b` does not match `jle`). The list never contained the mnemonic the function actually uses.
- **Defect 2 ŌĆö leaky extraction:** `awk '/<_ZL13sum_threshold[^_]/,/^$/'` re-opens the range on every line *referencing* the symbol (call sites, `lea` comment annotations), sweeping unrelated code chunks into the extraction. The old committed disasm file (465 lines) contains such swept-in code; the guard historically passed because a `jl`-family instruction existed somewhere in that noise ŌĆö a false pass. The new build's swept-in chunks contain none, exposing Defect 1.
- **Demo 02 (`check_volatile_codegen.sh`):** `grep -A 40 '<_ZL9worker_fn' | grep -c 'movsd'` has (a) the same reference-line leak ŌĆö every call site / address comment matching the symbol contributes a 40-line window of unrelated code to the count, so a stray `movsd` elsewhere inflates COUNT (false-pass vector); (b) a fixed 40-line window that can miss the genuine load+store pair if entry padding grows ŌĆö and the new toolchain adds `endbr64` plus larger nop sleds, making this false-fail more likely now; (c) the bare substring `movsd` accidentally also matches `vmovsd` (the AVX encoding) ŌĆö that tolerance is correct and must be preserved deliberately in the fix.

## Tasks

### Task 1 ŌĆö anchor the branchy extraction to the function header
objdump function headers start at column 0 and end in `>:`; reference lines are indented and end in `>`. Find:
```
        | awk '/<_ZL13sum_threshold[^_]/,/^$/' \
```
Replace with:
```
        | awk '/^[0-9a-f]+ <_ZL13sum_threshold[^_].*>:/,/^$/' \
```

### Task 2 ŌĆö correct the mnemonic check to the comparison family
Find:
```
    if ! grep -qE '\b(jl|jge|jb|jae)\b' "${DISASM_OUT}"; then
```
Replace with:
```
    if ! grep -qE '\bj(l|le|g|ge|b|be|a|ae)\b' "${DISASM_OUT}"; then
```
Deliberately **exclude `je`/`jne`** ŌĆö those are loop-control/emptiness branches present even in branchless code; including them would make the check vacuous. The set above is the signed/unsigned comparison family, which is what a data-dependent threshold branch compiles to (`jle` today; tuning may flip it to `jl`/`jge` etc. on other compiler versions ŌĆö all acceptable).

Update the two echo strings that name the old list (`must contain jl/jge` at ~line 604 and `no jl/jge/jb/jae found` at ~609) to say `a conditional comparison branch (jl/jle/jg/jge/jb/jbe/ja/jae)`.

### Task 3 ŌĆö anchor the branchless extraction the same way
Find:
```
        | awk '/<_ZL24sum_threshold_branchless/,/^$/' 2>/dev/null || true)
```
Replace with:
```
        | awk '/^[0-9a-f]+ <_ZL24sum_threshold_branchless.*>:/,/^$/' 2>/dev/null || true)
```
(Same leak: its cmov-present / SIMD-absent checks could otherwise false-pass or false-fail on swept-in neighbouring code.)

### Task 4 ŌĆö regenerate and sanity-check the committed disasm artifact
Run the guard path once (`run_one.sh 01-branch-prediction` up to/through the disassembly step, or extract equivalently) so `site/src/data/perf/01-branch-prediction.disasm.txt` is regenerated by the fixed extraction. The regenerated file legitimately differs from the committed one (`endbr64`, padding, addresses ŌĆö cosmetic toolchain drift; the loop body is identical).

### Task 5 ŌĆö fix the demo-02 volatile-codegen guard (`bench/scripts/check_volatile_codegen.sh`)
Replace the windowed substring count with an anchored full-function extraction and a word-anchored count that explicitly accepts both SSE2 and AVX encodings. Find:
```
COUNT=$(objdump -d "$BINARY" \
    | grep -A 40 '<_ZL9worker_fn' \
    | grep -c 'movsd' || true)
```
Replace with:
```
COUNT=$(objdump -d "$BINARY" \
    | awk '/^[0-9a-f]+ <_ZL9worker_fn.*>:/,/^$/' \
    | grep -cE '\bv?movsd\b' || true)
```
This counts `movsd`/`vmovsd` within the entire `worker_fn` body only ŌĆö no reference-line windows, no 40-line cap. Keep the `>= 2` threshold and the failure messaging; update the OK/FAIL strings to say `movsd/vmovsd`. (`_ZL9worker_fn` is length-prefix-anchored by mangling, so no longer-name collision; the header anchor handles reference lines.)
