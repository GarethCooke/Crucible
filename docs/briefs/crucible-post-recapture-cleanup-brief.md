# Crucible — post-recapture branch cleanup (verify tally + demo-03 asm tracking)

CC brief, branch `feature/boost-off-recapture`. Two small cleanups surfaced during the boost-off recapture. Both are branch hygiene — independent of the recaptured JSON data commit and of the MDX re-derivation.

## Context

- The recapture is complete; all nine Machine 1 JSONs pass `verify_boost_off.sh` (`turbo=false, turbo_source=cpb, freq_max_available=3900`).
- Two loose ends: (1) `verify_boost_off.sh` errors on its final tally line despite every demo passing; (2) four generated `.s` files under demo 03 showed up as modified by the rebuild, and their tracking status needs a decision — they may be displayed content (keep) or pure build artifacts (untrack).

## Task 1 — fix the `verify_boost_off.sh` tally bug

Running the script with no args prints all PASS lines, then fails with:
```
line 133: ${#MISSING[@]:-0}: bad substitution
```
`${#arr[@]:-0}` is invalid — the `#` (length) and `:-` (default) forms cannot combine in one expansion. This makes the script exit non-zero even when the whole corpus passes.

Fix:
1. Declare the array unconditionally near the top of the file (before the args/no-args branch), so it is always defined under `set -u`:
```
MISSING=()
```
2. On ~line 133, replace `${#MISSING[@]:-0}` with `${#MISSING[@]}`.

Result: with all demos passing and no missing files, the script exits 0.

## Task 2 — resolve demo-03 `.s` tracking (investigate first, then act)

These four files are git-tracked and were regenerated by the rebuild:
```
bench/demos/03-simd-blackscholes/asm/scalar_poly.s
bench/demos/03-simd-blackscholes/asm/scalar_libm.s
bench/demos/03-simd-blackscholes/asm/sse2_intrinsics.s
bench/demos/03-simd-blackscholes/asm/avx2_fma_intrinsics.s
```
**Do not blindly gitignore them.** Demo 03 is the SIMD post; these may be *displayed content* (like demo 01's committed `01-branch-prediction.disasm.txt`), in which case they belong in the repo. Determine which:

1. Check whether the site references them:
```
grep -rn "asm/\|\.s\"\|scalar_poly\|sse2_intrinsics\|avx2_fma" site/src/
```
and inspect demo 03's post component / MDX for any import or render of these files (a `<CodeCompare>` or similar reading from `asm/`). Also check the demo-03 README / any build step.

2. **If referenced (displayed content):** keep them tracked. Commit the regenerated versions. Add a note to the demo-03 re-derivation items that any asm quoted in prose must be re-checked against the regenerated `.s` — though instruction sequences are unchanged by boost state, so expect only cosmetic diffs (`endbr64`, `.cfi`/label noise from the toolchain). Do **not** gitignore.

3. **If not referenced (pure build artifacts):** add a pattern to `.gitignore` covering generated asm (e.g. `bench/demos/**/asm/*.s`), `git rm --cached` the four files (keep the working-tree copies), and commit as a standalone "stop tracking generated asm" change.

Either way, these `.s` files must **not** be staged into the recapture JSON data commit.
